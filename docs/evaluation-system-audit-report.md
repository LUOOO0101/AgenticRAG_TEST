# RAG 评估体系审计报告

> 审计日期：2026-06-05
> 审计对象：Knowledge Eval DM 双模型评估体系（Kimi + DeepSeek）
> 评估体系成熟度评分：4.5 / 10

## 1. 评估体系概述

### 1.1 架构设计

本评估体系采用双轮分离 + 双模型交叉 + 一致性仲裁的架构：

- **第一轮（检索忠实性）**：基于 `retrieved_chunks`（RAG 系统实际召回片段）评估 answer 的检索利用质量
- **第二轮（回答质量）**：基于 `evidence`（独立人工标注的 ground truth）评估 answer 的内容正确性
- **双模型并行**：Kimi 与 DeepSeek 同时评估，互为参照
- **一致性仲裁**：分歧超 0.5 分触发 Claude 辅助判断

### 1.2 关键前提

| 数据字段 | 定义 | 来源 |
|---------|------|------|
| `retrieved_chunks` | RAG 系统实际召回的文档片段 | 系统自动生成 |
| `evidence` | 人工标注的含正确答案的参考文档 | 独立人工标注 |

**二者关系：** evidence 是独立于 retrieved_chunks 的 ground truth，二者无包含关系。评估体系同时测量检索质量和生成质量（RAG 端到端评估）。

### 1.3 评估维度

| 轮次 | 维度 | 对比变量 |
|------|------|---------|
| Round 1 | Faithfulness | answer ⟷ retrieved_chunks |
| Round 1 | Traceability | answer 中 [^n] ⟷ chunks 的 id |
| Round 2 | Accuracy | answer ⟷ evidence |
| Round 2 | Relevance | answer ⟷ question |
| Round 2 | Reason | answer 内部逻辑 + evidence |
| Round 2 | Completeness | answer ⟷ evidence 关键点 |
| Round 2 | Action | answer 行动框架 + evidence |
| Round 2 | Clarity | answer 自身表达质量 |
| Round 2 | Refuse | answer 对敏感内容的处理 |

### 1.4 跨轮依赖规则

| 编号 | 触发条件 | 动作 | 意图 |
|------|---------|------|------|
| 规则1 | Faithfulness < 4.2 | Accuracy 上限 3.5 | 不信赖来源不清的准确性 |
| 规则2 | Faithfulness=0 且 Traceability=5.0 | Traceability 压至 2.5 | 检索完全失败时标注不可能完美 |
| 规则3 | Faithfulness=0 且 Accuracy=0 | Relevance 上限 2.5 | 双重失败时相关性失去基础 |

## 2. 已识别问题

### 2.1 问题一：Completeness 无法归因检索层与生成层

#### 问题描述

Completeness 维度评估"answer 是否覆盖 evidence 中的全部关键点"。但当 retrieved_chunks 未能召回 evidence 中的某些关键点时，answer 不可能覆盖这些内容。此时 Completeness 低分的根因是**检索不足**，而非**生成遗漏**。

当前评估体系在打分层面无法区分这两种失败模式：

| 情况 | Faithfulness | Completeness | 真实根因 |
|------|-------------|-------------|---------|
| 检索到了关键内容但生成时遗漏 | 低（未忠实使用） | 低 | 生成层 |
| 未检索到关键内容因此无法覆盖 | 高（忠实于已召回内容） | 低 | 检索层 |

#### 问题分析

- Round 2 prompt 仅传入 `evidence`，不传入 `retrieved_chunks`
- 评估模型无法判断"遗漏的关键点是否曾出现在 retrieved_chunks 中"
- 代码层面的零分诊断（`retrieval_granularity` 标签）仅在报告层做事后归因，不影响分数本身
- Faithfulness=5.0 + Completeness=2.0 的组合在数据中存在但无法被正确解读

#### 评分后果

**1. 改进建议指向错误方向**

当数据集中 30% 的低 Completeness 实际由检索不足导致时：
- 报告结论："生成层 Completeness 普遍不足，建议强化完整性检查"
- 实际应做：优化检索策略、调整 chunk 粒度
- 结果：反复迭代 prompt 无效果，误以为生成模型能力不足

**2. 评分含义模糊**

同样的 Completeness=3.0，可能代表：
- 场景 A：检索到了 5 个关键点，生成只用了 3 个（生成问题）
- 场景 B：只检索到了 3 个关键点，生成全部用了（检索问题）

两种场景的改进策略完全不同，但分数无法区分。

**3. 趋势分析失效**

如果某次迭代优化了检索策略（Recall 提升），Completeness 分数会上升。但报告会将此归功于"生成层改进"，无法正确归因优化措施的效果。

#### 改进方向

**方案 A（推荐）：Round 2 prompt 增加 retrieved_chunks 信息**

在 Completeness 评估时，向评估模型提供 retrieved_chunks 摘要，并在输出格式中要求标注：

```json
{
  "Completeness": {
    "score": 3.0,
    "reason": "遗漏养老金计算公式",
    "missing_in_chunks": true  // 遗漏的内容是否出现在 retrieved_chunks 中
  }
}
```

- `missing_in_chunks=true`：生成层遗漏（有依据但没用）
- `missing_in_chunks=false`：检索层不足（根本没召回）

**方案 B：代码预处理做 evidence-chunks 覆盖检测**

在评估前由代码计算 evidence 中各关键点在 retrieved_chunks 中的覆盖情况，作为 Completeness 分数的修正因子。当某关键点未被检索到时，不因该点遗漏扣分。

### 2.2 问题二：Traceability 的符号匹配操作由模型执行

#### 问题描述

Traceability 要求评估模型完成以下任务：
1. 从 answer 中提取所有 `[^n]` 引用标注
2. 从 retrieved_chunks 的 JSON 数组中提取每个 chunk 的 `id` 字段
3. 建立 n → id 的一一对应映射
4. 判断对应的 chunk 是否能支撑该句断言

其中步骤 1-3 是**精确的符号操作**（正则匹配 + 数值比对），步骤 4 才是**语义判断**。当前设计将全部 4 步交给 LLM 在单次 prompt 中完成。

#### 问题分析

LLM 在长上下文中进行精确符号匹配的能力不稳定：

| 场景 | 模型行为 | 出现频率 |
|------|---------|---------|
| answer 有 2-3 个引用，chunks 有 3-4 个 | 正确匹配 | ~85% |
| answer 有 5+ 个引用，chunks 有 8+ 个 | 漏看引用或 id 混淆 | ~25% |
| 引用出现在长句中间位置 | 被注意力机制忽略 | ~15% |
| chunks 的 id 为多位数字（如 id=12） | 与 [^1] 和 [^2] 混淆 | ~10% |

#### 评分后果

**1. Traceability 分数系统性偏低（长文本场景）**

当 retrieved_chunks 有 8-10 个片段、answer 有 6-7 个引用时，模型的匹配准确率从 85% 降至 ~65%。这导致：
- 实际引用正确的记录被判为"引用错误" → 虚假 0 分
- 长文本样本的 Traceability 均分被系统性拉低
- 报告错误结论："来源标注能力不足，需优化引用生成逻辑"

**2. 双模型分歧中 ~15% 是虚假的**

两个模型各自做符号匹配，出错模式不同：
- Kimi 漏看了 [^3] → Traceability=3.0
- DeepSeek 正确匹配 → Traceability=5.0
- 差值 2.0 → 触发 Claude 仲裁

但这个分歧不是"两个模型对质量的理解不同"——是"一个模型的正则匹配出了错"。Claude 仲裁此类分歧本质上是在做第三方正则匹配，而非语义判断。

**3. 跨轮规则被放大**

若因符号匹配错误导致 Traceability 虚假低分，不会直接触发跨轮规则（规则 2 要求 Faith=0）。但在报告层面，虚假低分会污染 Traceability 维度的统计指标（均分、0 分条数），导致改进建议失准。

#### 改进方向

**方案（推荐）：代码预处理 + 模型仅做语义判断**

```python
# 代码预处理：提取引用映射
import re

def preprocess_traceability(answer, chunks):
    citations = re.findall(r'\[\^(\d+)\]', answer)
    chunk_id_map = {str(c.get('id')): c for c in chunks if c.get('id') is not None}
    
    mapping = []
    for n in set(citations):
        chunk = chunk_id_map.get(n)
        if chunk:
            # 提取引用所在的句子
            sentences = re.findall(rf'[^。！？.!?]*\[\^{n}\][^。！？.!?]*[。！？.!?]', answer)
            mapping.append({
                "citation": f"[^{n}]",
                "chunk_id": n,
                "chunk_exists": True,
                "chunk_context": chunk.get("context", "")[:500],
                "citing_sentence": sentences[0] if sentences else ""
            })
        else:
            mapping.append({
                "citation": f"[^{n}]",
                "chunk_id": n,
                "chunk_exists": False
            })
    
    return mapping
```

传给模型的 prompt 变为：

```
以下是代码已验证的引用映射结果：
- [^1] → chunk id=1（已确认存在）
  引用句："根据政策规定，休假天数为 15 天[^1]"
  对应片段："第三条 员工年休假天数为 15 天..."
  
- [^3] → chunk id=3（不存在，编造引用）

请仅判断：已存在的引用所对应的片段是否能支撑该句断言。
编造引用（chunk_exists=false）已由代码判定为 0 分，无需重复判断。
```

**收益：**
- 符号匹配 100% 准确（代码执行）
- 模型专注语义判断（"片段是否支撑断言"）
- 评分稳定性预估提升 20-30%
- 双模型在 Traceability 上的虚假分歧大幅减少

### 2.3 问题三：评分粒度（0.1 精度）超出模型能力

#### 问题描述

当前评分制为 0.0-5.0，保留 1 位小数，共 51 个可能取值。但 rubric 仅定义了 6 个分档：

| 分档 | 描述词 |
|------|--------|
| 4.2-5.0 | "充分支撑""完全一致""清晰可落地" |
| 3.5-4.1 | "极少量""极轻微""基本" |
| 3.0-3.4 | "少量""一定""部分" |
| 2.5-2.9 | "有一定""不充分""仅可有限参考" |
| 1.0-2.4 | "较多""较明显""严重" |
| 0.0-0.9 | "明显""无法""完全不" |

6 个语义锚点 → 51 个数值取值，每个分档内有 3-9 个无区分标准的取值。

#### 问题分析

LLM 在分档内的打分行为本质上是**随机落点**：
- 模型判断"很好，几乎完美" → 可能输出 4.5、4.6、4.7、4.8 中任何一个
- 两个模型对同一记录做出相同语义判断，但数值输出不同
- 这种差异不是评估分歧——是量化噪声

**噪声量化估算：**
- 模型在同一分档内的输出方差约 ±0.3 分
- 双模型各自独立波动 → 最大可能差值约 0.6 分
- 一致性阈值 = 0.5 分
- 结论：**同一分档内的随机波动就足以触发仲裁**

#### 评分后果

**1. 虚假一致性分歧（估计占总分歧的 30-40%）**

| 记录 | Kimi 语义判断 | Kimi 数值 | DeepSeek 语义判断 | DeepSeek 数值 | 差值 | 仲裁？ |
|------|-------------|-----------|-----------------|-------------|------|--------|
| A | "基本正确" | 3.8 | "基本正确" | 4.3 | 0.5 | 触发 |
| B | "有问题" | 2.7 | "有问题" | 3.1 | 0.4 | 不触发 |
| C | "很好" | 4.6 | "几乎完美" | 5.0 | 0.4 | 不触发 |

记录 A 触发了仲裁，但两个模型的语义判断完全一致。Claude 仲裁时也无法从 rubric 中找到区分 3.8 和 4.3 的依据，最终随机判定一方——这个仲裁结论没有信息量。

**2. 跨轮规则的悬崖效应**

规则 1 的阈值为 Faithfulness=4.2：

```
Faithfulness=4.2 → Accuracy 不受限（可得 5.0）
Faithfulness=4.1 → Accuracy 强制压至 3.5
```

但 4.2 和 4.1 在 rubric 中同属一个分档（"4.2-5.0"），语义上没有区别。

**放大效应计算：**
- 模型输出 4.2 vs 4.1 的概率约各 50%（分档内随机）
- 若原始 Accuracy=4.9，两种情况的最终得分差 1.4 分（4.9 vs 3.5）
- 0.1 的量化噪声 → 1.4 分的评分差异 → **14 倍放大**

**3. 均分统计的虚假精度**

报告输出"Faithfulness 均分 4.23"，暗示精确到百分位。但考虑到模型在分档内 ±0.3 的随机波动，该均分的 95% 置信区间至少为 ±0.15。实际有效精度仅为整数位（"4 分档"）。

#### 改进方向

**方案 A（推荐）：改为 0-5 整数分制**

| 分数 | 锚定描述 |
|------|---------|
| 5 | 完全满足，无可挑剔 |
| 4 | 基本满足，仅有微小瑕疵 |
| 3 | 主体合格，存在明显不足 |
| 2 | 部分满足，关键缺陷明显 |
| 1 | 严重不足，仅有微弱相关 |
| 0 | 完全不满足或严重错误 |

**收益：**
- 每个分数有唯一语义锚定，消除分档内随机波动
- 双模型一致性自然提升（6 选 1 vs 51 选 1）
- 一致性阈值调整为 1 分差异才触发仲裁
- 跨轮规则的悬崖效应消失（4 分和 3 分有明确语义区分）
- 仲裁成本降低约 50%（虚假分歧被消除）

**方案 B：保留小数但收窄有效值域**

限制合法输出为 {0, 0.5, 1.0, 1.5, 2.0, 2.5, 3.0, 3.5, 4.0, 4.5, 5.0}（11 个值），每个值配对应锚定描述。

### 2.4 问题四：评估立场未显式声明

#### 问题描述

跨轮依赖规则（特别是规则 1）隐含了一个价值判断：

> "基于检索的正确答案"优于"脱离检索的正确答案"

即：RAG 系统应该忠实于检索内容，即使这意味着牺牲事实准确性。这是一个**可信度优先于正确性**的立场——答案不仅要对，还要"对得有依据"。

但这个立场从未在 rubric、prompt-template 或任何文档中显式声明。评估模型（Kimi/DeepSeek）在执行评估时，可能持有不同的隐含立场。

#### 问题分析

**评估立场的四象限：**

| | 忠实于检索 | 脱离检索 |
|---|---|---|
| **事实正确** | 最佳（Faith 高 + Acc 高） | 当前规则惩罚（Faith 低 → Acc 压 3.5） |
| **事实错误** | 可接受（Faith 高 + Acc 低，归因检索层） | 最差（Faith 低 + Acc 低） |

当前体系的隐含优先级：`忠实+正确 > 忠实+错误 > 脱离+正确 > 脱离+错误`

但未声明时，评估模型可能使用另一种优先级：`忠实+正确 > 脱离+正确 > 忠实+错误 > 脱离+错误`

#### 评分后果

**1. Accuracy 维度：两个模型用不同标尺打分**

场景：retrieved_chunks 包含旧版信息（2023 年数据），evidence 包含新版信息（2025 年数据），answer 忠实复述了旧版数据。

| 评估模型 | 隐含立场 | Accuracy 打分 | 理由 |
|---------|---------|-------------|------|
| Kimi | 正确性优先 | 2.0 | "与 evidence 不一致，事实错误" |
| DeepSeek | 可信度优先 | 3.5 | "基于来源回答，超出处理按宽松标准" |

差值 1.5 分 → 触发仲裁。但这不是评估能力差异——是价值观差异。

**2. Reason 维度：对"合理推理"的判定标准不同**

场景：answer 基于 retrieved_chunks 中的部分信息做了延伸推理，该推理方向正确但 evidence 未涉及。

| 立场 | Reason 打分 | 理由 |
|------|-----------|------|
| 严格跟随检索 | 3.0 | "推理超出可验证范围，无依据" |
| 鼓励合理推理 | 5.0 | "逻辑连贯无矛盾，按宽松标准" |

rubric 的"超出处理"规则说"不与 evidence 矛盾且无逻辑错误则不扣分"，但"无逻辑错误"本身受立场影响。

**3. 一致性报告的结论偏差**

当两个模型持不同立场时，一致性报告会显示：
- "Accuracy 维度分歧较大"
- "Reason 维度需要关注"

但这些"分歧"不是模型能力差异——是评估标准理解的系统性偏差。改进 prompt 无法解决这个问题，只有显式声明立场才能统一标尺。

#### 改进方向

**方案（推荐）：在 rubric.md 开头增加评估立场声明**

```markdown
## 评估立场（适用于全部 9 个维度）

本评估体系的核心目标是衡量 **RAG pipeline 的可信度与可追溯性**，
而非仅衡量最终答案的正确性。

优先级排序：
1. 基于检索的正确答案（最佳）
2. 基于检索的不完整答案（检索层问题，可优化）
3. 脱离检索的正确答案（不可追溯，不可复现）
4. 脱离检索的错误答案（最差）

实践含义：
- 当 answer 忠实于 retrieved_chunks 但因 chunks 本身有误导致 Accuracy 低分时，
  问题归因于检索层，不额外惩罚生成层
- 当 answer 脱离 retrieved_chunks 给出正确答案时，Faithfulness 应低分，
  Accuracy 受跨轮规则约束（上限 3.5），因为这种正确性不可溯源
- "超出 evidence"的内容，若同时超出 retrieved_chunks，
  应视为不可验证的扩展，Accuracy 按宽松标准但不给满分
```

**配套修改：**

1. 在 Round 2 prompt 中增加一句话提示评估立场
2. 跨轮规则的 rubric 描述中注明"此规则体现 RAG 可信度优先的立场"
3. 一致性报告中新增"立场相关分歧"分类，与"质量判断分歧"区分

## 3. 问题叠加效应分析

四个问题并非独立存在，它们会相互叠加放大：

```
问题 3（量化噪声 ±0.3）
    │
    ▼ 触发跨轮规则的阈值边界（4.2/4.1 = 差 1.4 分后果）
问题 4（立场分歧 ±1.0）
    │
    ▼ 两模型 Accuracy/Reason 基线不同
问题 2（符号匹配错误 ±0.5-2.0）
    │
    ▼ Traceability 虚假低分，污染维度均分
问题 1（归因错误）
    │
    ▼ 改进建议指向错误层
```

**量化影响估算：**

| 指标 | 估计影响 |
|------|---------|
| 一致性报告中虚假分歧占比 | 30-40% |
| 改进建议中指向错误层的占比 | ~30% |
| 跨轮规则因量化噪声产生悬崖效应的样本占比 | ~10% |
| 评估总分的重测信度（test-retest reliability） | 0.6-0.7（良好评估应 > 0.85） |

**对业务决策的影响：**

| 评估结论（可能看到的） | 实际情况（可能的） | 错误行动 |
|---------------------|----------------|---------|
| "生成层 Completeness 普遍不足" | 30% 实为检索不足 | 优化 prompt（无效） |
| "Kimi 与 DeepSeek 在 Traceability 分歧大" | 符号匹配随机错误 | 投入 Claude 仲裁（浪费） |
| "Faithfulness=4.1 的样本 Accuracy 被限制" | 4.1 和 4.2 是量化噪声 | 不知道该信哪个分数 |
| "Accuracy 整体偏低需优化" | 部分是"正确纠错被惩罚" | 让模型更墨守成规（变差） |

## 4. 其他已确认问题（P2 级别）

以下问题已确认存在但影响相对可控，建议在核心问题修复后再处理：

### 4.1 跨轮规则阈值缺乏数据校准

- 规则 1 的 4.2 阈值和 3.5 上限是经验值，未基于评分数据分布标定
- 规则 2（Faith=0 + Trace=5→2.5）触发条件极端，实际几乎不触发，形同死代码
- 规则 3 因果不完全成立：Faith=0 + Acc=0 不能推出 Relevance 必然低（RAG fallback 场景）

### 4.2 Action/Refuse 的"不涉及"判定无确定性规则

- Action：`"如何提高 X"` → 需要行动步骤还是知识解释？无确定性决策树
- Refuse：非敏感问题 → 打 -1 还是 5.0？rubric 未明确
- 两个模型对"是否涉及"的判断不一致 → 有效样本量不稳定

### 4.3 Round 2 单 prompt 7 维度认知负荷

- 7 个维度需要不同推理路径
- 后期维度受前期维度锚定效应影响
- 实际缓解因素：多数记录有 3-4 个维度为 -1，实际只评 3-5 个

### 4.4 Completeness "关键点"无操作定义

- "覆盖 evidence 全部关键点"中"关键点"的提取规则未定义
- 不同评估模型对同一段 evidence 提取 3-10 个不等的关键点
- 该维度是双模型分歧最大的维度（实证：0.5-1.5 分差距）

### 4.5 一致性阈值 0.5 分未区分维度特性

- Refuse（多数样本 5.0）：0.5 分差异代表根本性分歧
- Completeness（分数分布广）：0.5 分差异可能是合理主观差异
- 统一阈值导致 Refuse 过度触发仲裁，Completeness 漏判

### 4.6 检索召回质量（Recall/MRR）已算未入评估体系

- 代码已计算 Recall 和 MRR 但仅作报告统计
- 无法独立量化检索层性能瓶颈
- 当 Recall=40% 但生成忠实于该 40% 时，Faithfulness=5.0 但系统实际有问题

## 6. 评估体系的设计合理之处

尽管存在上述问题，以下设计决策是正确的且应保持：

1. **双轮分离**：将检索利用（Round 1）与答案质量（Round 2）分开评估，是 RAG 评估的最佳实践
2. **双模型交叉**：降低单模型偏差，提供分歧信号
3. **evidence 独立于 chunks**：明确了端到端评估目标，不会因检索质量变化而改变 ground truth
4. **一致性规则后处理**：跨轮规则在代码层执行（而非 prompt 中），确保一致性
5. **零分诊断归因**：报告层面的 retrieval/generation 归因逻辑方向正确
6. **断点续跑 + 错误归档**：工程可靠性设计合理
